# 02 · 제품 스펙 (이번 스프린트 구현 범위)

> 이 문서는 **좁게** 쓴다. 이번에 실제로 구현할 화면만 담는다.
> 범위 = **① 체크리스트 + ② 백과.** 청소·로그인·백엔드·저장 기능은 여기 없다(`05-out-of-scope.md`).
> 데이터는 전부 하드코딩 더미(`03-data.md`).

---

## 0. 정보 구조 (라우트)

```
/                      홈 (두 기능 진입)
/checklist             방 구할 때 체크리스트 (상황 선택)
/checklist/:situation  선택한 상황의 단계별 체크리스트
/wiki                  자취 백과 (카테고리 그리드 + 검색)
/wiki/:category        카테고리별 항목 목록
/wiki/entry/:id        개별 항목 상세 (3줄 요약 + 링크아웃)
```

라우팅은 클라이언트 사이드(react-router). 백엔드 호출 없음.

---

## 1. 홈 `/`

- 상단: `SERVICE_NAME`(상수) + 한 줄 소개.
- 두 개의 큰 진입 카드:
  - **"방 구하는 중이에요"** → `/checklist`
  - **"이미 자취 중인데 뭔가 막혔어요"** → `/wiki`
- 하단: 백과 검색창(전역) — 홈에서도 바로 키워드 검색 가능.
- ⚠️ 청소 카드는 넣지 않는다. (파킹)

> 설계 의도: 두 기능의 **상호작용 방식이 다름**(의사결정 지원 vs 반응형 조회)을 진입 문구로 드러낸다.

---

## 2. 체크리스트 `/checklist` (H1)

### 2-1. 상황 선택 `/checklist`
- 방 유형 선택: `원룸`, `오피스텔`, `반지하/1층`, `옥탑`, `기타`.
- (선택) 상황 태그: `자연광 중요`, `저소음 중요`, `반려동물` 등 — 선택 시 관련 항목 강조.
- "체크리스트 시작" → `/checklist/:situation`

### 2-2. 단계별 체크리스트 `/checklist/:situation`
**순서가 핵심이다.** 사용자가 실제로 방을 구하는 순서대로 단계를 제시한다.

```
STEP 1 · 방문 전 (동네·입지)
  - 치안 (밤에 걸어봤나, 가로등)
  - 오르막길 / 언덕
  - 주변 상권 (편의점·마트 거리)
  - 대로변 소음 (창문이 큰길 쪽인가)
  - 코인 세탁방 거리 + 빨래 널 공간 크기

STEP 2 · 방문 시 (집 상태)
  - 채광 / 햇빛 드는 방향
  - 곰팡이 · 누수 흔적 (벽지 들뜸, 천장 얼룩)
  - 찍힘 · 하자 (사진으로 기록해두기)
  - 수압 (수도·샤워 틀어보기)
  - 기본 옵션 가구·가전 목록 확인
  - 벌레 흔적 · 방충 상태
  - 창문 방향 / 환기

STEP 3 · 계약 시
  - 계약서 특약 확인
  - 세탁기·에어컨 청소·관리비 책임 명시 여부
  - 등기부등본 확인
  - 청년월세 지원 대상 여부

STEP 4 · 계약 후 (입주 준비)
  - 전입신고 + 확정일자
  - 전기·가스 명의 변경
```

- 각 항목: **체크박스 + "왜 확인해야 하나" 한 줄 툴팁.**
  (툴팁이 곧 "모르는 걸 모름" 해결 지점. 인터뷰에서 이 툴팁 반응을 본다.)
- 체크 상태는 **세션 메모리(React state)만**. 저장·로그인 없음.
- 하단: "빠진 항목 제보" 텍스트 입력 (전송 안 함, UI만 — 인터뷰에서 반응 관찰용).

> 인터뷰 훅: 이 화면을 보여주고 "여기서 빠진 거 / 순서 이상한 거 있어요?"를 묻는다(`01` 파트 A).

## 3. 자취 백과 `/wiki` (H2)

> 백과의 정체 = **콘텐츠 라이브러리가 아니라 큐레이션 지도.**
> 각 항목은 "목차 + 3줄 요약 + 검증된 최고 자료 링크"다. 물리 작업은 텍스트로 정면 설명하지 않고 링크아웃한다.

### 3-1. 카테고리 그리드 `/wiki`
- 상단: **전역 검색창** (키워드로 항목 바로 찾기 — "조립 시간 죽이기"의 핵심 UX).
- 카테고리 카드 그리드:
  - `생활 수리` (변기 막힘, 우풍/문풍지, 못질, 현관 잠금장치)
  - `가전 관리` (세탁기 물샘, 에어컨 콘센트 전압, 렌지후드)
  - `분리수거 · 쓰레기`
  - `세탁` (색깔별·속옷별)
  - `식생활` (가성비 식단·밀키트, 냉장고·식자재 관리)
  - `행정` (전입신고, 청년월세, 전기·가스 납부)
  - `계절` (습도·결로, 벌레 대비)
  - `안전` (현관 걸쇠·가스켓)

### 3-2. 카테고리별 목록 `/wiki/:category`
- 해당 카테고리의 항목 리스트.
- 각 항목 행: 제목 + "언제 찾게 되나(트리거)" 한 줄 + 태그.

### 3-3. 항목 상세 `/wiki/entry/:id`
각 항목의 표준 구조 (`03-data.md` 스키마와 일치):
```
제목
언제 찾게 되나 (트리거 상황 1줄)
3줄 요약 (핵심만)
[검증된 자료 보기] → 외부 유튜브/블로그 링크아웃
관련 항목 (같은 카테고리 2–3개)
```
- **3줄 요약 + 링크아웃**이 원칙. 긴 본문·단계별 텍스트 설명을 넣지 않는다(영상에 짐).
- 상세 화면에 **"이거 유용하면 저장"** 버튼을 둔다 (실제 저장 안 함 — 인터뷰에서 **북마크 행동 관찰용**, `01` 파트 B 10번).

> 인터뷰 훅: 특정 항목(예: 변기 막힘)을 보여주고
> "이거 있었으면 이걸 썼을 것 같아요, 아니면 유튜브 쳤을 것 같아요?"를 묻는다(`01` 파트 B).

---

## 4. 공통 UX 원칙

- 모바일 우선 (첫 자취생은 방 보러 다니며 폰으로 쓴다).
- 텍스트는 짧게, 스캔 가능하게. 백과는 "읽는 문서"가 아니라 "훑는 지도".
- 로딩·빈 상태·에러 상태는 최소한만 (더미 데이터라 실패 케이스 없음).

---

## 5. 이번 스펙에 **없는** 것 (재확인)

- 청소 기능(루틴/스케줄러/도구) — 파킹
- 로그인·회원·저장·북마크 실제 동작 — UI 흉내만
- 백엔드·DB·API — 전부 하드코딩 더미
- 커뮤니티·후기·업체 연결·지도 임베드 — 이번 범위 아님

자세한 제외 사유는 `05-out-of-scope.md`.
