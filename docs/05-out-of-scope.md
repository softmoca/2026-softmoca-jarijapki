# 05 · 범위 밖 (제외 목록 · 파킹 · 자기진단)

> 이 문서의 목적: **검증 안 된 것을 미리 만드는 낭비를 막는다.**
> 화요일 문서 경고 — "AI는 구현을 탁월하게 해주는데, 신나서 만들다 보면 아무도 안 쓸 걸 탁월하게 구현하고 있다."

## 1. 이번 스프린트에서 **구현하지 않는** 것

### 청소 기능 (파킹 — 가장 중요)
- 청소는 비전에 **있다**(`00-context.md`). 하지만 이번 스프린트엔 **구현하지 않는다.**
- 이유:
  - 청소는 "읽는 레퍼런스"가 아니라 **"하는 도구/루틴"**이라 상호작용 방식이 다르다 → 별도 검증 필요.
  - 형태(루틴/스케줄러 vs how-to)가 아직 미검증(H3).
  - 별도 컨텍스트가 많아, 확보 후 독립 문서로 설계 예정.
- ⚠️ 주의: 백과 안에 "에어컨 청소 팁" 같은 **청소 관련 백과 항목**은 있을 수 있다(링크아웃 레퍼런스).
  하지만 **청소 전용 기능(루틴/스케줄/도구)**은 별개이며 이번 범위 밖이다. 이 둘을 혼동하지 않는다.

### 기타 제외
- 로그인·회원가입·사용자 계정
- 실제 저장/북마크 동작 (UI 흉내만 — 인터뷰 행동 관찰용)
- 백엔드·DB·API·크롤링
- 실제 매물 데이터, 지도 임베드, 좌표
- 업체 연결·예약·결제
- 커뮤니티·댓글·후기 작성
- 알림·푸시
- 다국어

## 2. 왜 백과는 "만들되" 청소는 "안 만드나"

셋 다 같은 뿌리의 페인이지만, **위험 가정이 다르고 검증 준비 상태가 다르다.**

| 기능 | 이번 스프린트 | 이유 |
| --- | --- | --- |
| 체크리스트 | ✅ 구현·검증 | 위험 가정(H1) 검증이 쉽고, 주 wedge |
| 백과 | ✅ 구현, 검증은 습관 프로브 위주 | 페인 검증됨(H0), 위험 가정(H2=습관)은 프로토타입으로 부분 확인 가능 |
| 청소 | ❌ 파킹 | 형태 미검증(H3), 별도 컨텍스트 필요 |

---

## 3. "확장은 존재한다"의 두 의미 구분

혼동 주의. 아래 둘은 다르다:

- **페인·비전 차원의 확장** (백과·청소가 붙을 것이다) → ✅ 존재한다. `00-context.md`에 적는다.
- **검증된 해결책 차원의 확장** (백과·청소가 *이런 형태로* 먹힌다) → ❌ 아직 아니다.
  백과의 습관·유입 가정, 청소의 형태 — 모두 미검증.

→ 그래서 아키텍처는 확장을 수용하되(폴더 자리만), **검증 안 된 화면은 미리 만들지 않는다.**

---

## 4. 자기진단 체크리스트 (화면·기능을 늘리고 싶을 때)

새 화면/기능을 추가하려는 충동이 들면, 아래를 통과하는지 스스로 묻는다:

- [ ] 이건 `02-product-spec.md`에 있는가? (없으면 멈춘다)
- [ ] 이걸로 확인하려는 **위험 가정**이 `01-validation-map.md`에 있는가?
- [ ] 이건 이번 인터뷰에서 **행동 신호(A/B)**를 끌어내는 데 필요한가, 아니면 "있으면 좋아서"인가?
- [ ] 이 물리 작업을 **텍스트로 정면 설명**하려 하고 있진 않은가? (그렇다면 링크아웃으로)
- [ ] 이건 청소 기능을 슬쩍 끌어들이는 건 아닌가? (그렇다면 파킹)
- [ ] 검증도 안 된 형태 위에 **확장 아키텍처를 미리 깔고** 있진 않은가?

하나라도 걸리면 멈추고 `01`·`02`를 다시 본다.

## 5. 위험 신호 (스스로 감지할 것)

- 인터뷰에서 "좋은 아이디어네요"(C급)만 모으고 있다 → 과거 행동을 못 물었다.
- 체크리스트와 백과를 한 화면에서 동시에 보여주고 있다 → 신호 오염(`01` 위반).
- 백과에서 만족도만 묻고 습관("유튜브 대신 이걸 쓸까")을 안 물었다 → H2 미검증.
- 프로토타입을 예쁘게 만드는 데 시간을 쓰고 있다 → 프로토타이핑(빠른 학습)이 아니라 프로토타입 미화.
